# 视频关联设计

## 背景

视频库需要支持一条视频关联另一条视频，用来记录同一内容在不同拍法、不同表达、不同平台合规结果下的复拍和变体关系。典型场景包括：

- 某条视频数据表现好，后续换一种形式再拍一遍。
- 某条视频违规，下架或受限后，用另一种表达方式重拍。
- 同一个主题做了多个版本，需要在回看数据时快速互相跳转。

当前系统已经有 `Video.topicId`、`Video.scriptId`、`Topic.linkedVideoId`、`Script.videoId` 等生产链关系，并且会在选题、逐字稿、视频之间同步标题。新的视频关联不应进入这条标题同步链，因为两个视频可能标题、形式、平台结果和数据都不同。

## 设计目标

- 支持任意两条视频建立关联。
- 建立关联时允许填写自由备注，不使用固定原因枚举。
- 在任一关联视频详情页都能看到对方。
- 不改变已有选题、逐字稿、视频之间的生产链语义。
- 删除视频时自动清理失效关联，避免详情页出现孤儿引用。
- 对现有数据迁移保持轻量，缺失新字段或新文件时按空关系处理。

## 推荐方案

新增独立关系集合 `videoRelations`，而不是把关系直接塞进 `Video`。

```ts
export interface VideoRelation {
  id: string
  fromVideoId: string
  toVideoId: string
  note?: string
  createdAt: string
  updatedAt: string
}

export interface AppData {
  // ...
  videoRelations: VideoRelation[]
}
```

关系在数据层以一条记录表达，展示层按 `fromVideoId === video.id || toVideoId === video.id` 查找，因此天然双向可见。备注只存一份，避免 A 详情页和 B 详情页出现两个不同备注。

## 备选方案

### 方案 A：`Video.relatedVideoIds?: string[]`

优点是实现最简单，改动面小。缺点是无法记录备注，不满足本次需求。

### 方案 B：`Video.relatedVideos?: { videoId; note }[]`

优点是仍然只改 `videos.json`，读写直观。缺点是如果双向各存一份，备注同步、删除清理、重复关系去重都会变复杂；如果只单向存，则另一条视频详情页不易自然展示。

### 方案 C：独立 `videoRelations`

优点是关系语义清晰、备注只有一份、未来可扩展关系类型或分析统计。缺点是需要新增类型、store action 和持久化文件。综合来看，这是最符合当前需求且长期更稳的方案。

## 交互设计

主要入口放在视频详情页，位置在“关联逐字稿”附近，新增“相关视频”区块。

区块内容：

- 已有关联以列表卡片展示。
- 每张卡片展示对方视频标题、状态、平台状态摘要、备注和创建时间。
- 卡片提供“查看”跳转和“移除关联”操作。
- 没有关联时显示简短空态和“关联视频”按钮。

新增关联流程：

1. 点击“关联视频”。
2. 弹出选择弹窗。
3. 支持按标题搜索视频。
4. 排除当前视频和已关联视频。
5. 选择目标视频。
6. 填写可选备注。
7. 保存后关系立即在两条视频详情页可见。

看板抽屉可以做只读摘要：如果当前视频有关联，展示最多 2 条相关视频标题和备注摘要，并提供“查看完整详情”承接编辑。视频列表页暂不新增列，避免表格继续变宽。

## Store 设计

新增 store action：

```ts
addVideoRelation: (fromVideoId: string, toVideoId: string, note?: string) => void
updateVideoRelation: (id: string, patch: Pick<VideoRelation, 'note'>) => void
deleteVideoRelation: (id: string) => void
```

约束：

- 不允许自己关联自己。
- 同一对视频只允许存在一条关系，方向无关。
- 新增和更新都刷新 `updatedAt`。
- 删除视频时，过滤掉所有包含该视频 ID 的 `videoRelations`。

## 持久化设计

多文件格式新增 `videoRelations.json`。读取时如果文件不存在，回退为空数组。写入时和其它数据文件一起保存。

旧 `data.json` 迁移到多文件格式时，如果没有 `videoRelations` 字段，也回退为空数组。默认数据可添加一条 demo 关系，也可以保持空数组；为了减少样例噪声，建议默认保持空。

## 验证计划

- 类型检查：`npx tsc -b --pretty false`
- 构建：`npm run build`
- store 级行为验证：
  - 新增 A-B 关系后，A、B 两边都能查到。
  - 再次新增 A-B 不产生重复关系。
  - A 不能关联 A。
  - 删除 A 后，关联 A 的关系被清理。
- UI 验证：
  - 视频详情页可新增关联并填写备注。
  - 关联卡片能跳转到另一条视频。
  - 移除关联后两边都不再显示。
  - 搜索弹窗不会列出当前视频和已关联视频。

## 风险与边界

- 关联备注不参与标题同步，也不影响选题和逐字稿。
- 暂不做关系类型枚举，避免过度设计。
- 暂不在视频列表页新增筛选或列展示，后续如果需要“查看所有重拍/变体组”，可基于 `videoRelations` 扩展分析视图。
- 暂不自动根据标题相似度或平台违规记录创建关联，所有关系由用户显式建立。
